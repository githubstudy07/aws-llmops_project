A案：シンプル・ユーザーフィードバック型（実運用重視）
機能ゴール: API Gateway 経由で「回答が役に立ったか（👍/👎）」というユーザーの直接的な評価を受け取り、Langfuse の Trace に user-feedback という名前で 0 または 1 のスコアを付与する。
非機能的目標: フロントエンド（Dify等）からの非同期・軽量なリクエスト処理。既存のチャットフローに影響を与えないエラーハンドリング。
B案：開発者マニュアル評価型（品質分析重視）
機能ゴール: ユーザーには意識させず、開発者が Langfuse のダッシュボード上で過去の Trace を確認し、「回答の正確性」や「トーン」を 0.0〜1.0 の範囲で手動採点（アノテーション）する。
非機能的目標: 評価基準（テンプレート）を定義し、採点の偏りを防ぐ。セキュリティ担当者のみが評価できる権限管理の意識。
C案：自動・品質ガードレール型（高度自動化）
機能ゴール: LLM (LLM-as-a-Judge) を使用して、生成された回答が「不適切な表現を含んでいないか」や「事実に基づいているか」を自動チェックし、スコアとして記録する。
非機能的目標: 評価用 LLM の API コスト抑制（サンプリング等）。評価ロジック自体のレイテンシ監視。


セキュリティ対策の選択肢
案1：ID の秘匿化（プロキシ ID 方式）
内容: Langfuse の生 ID ではなく、ランダムな一時的なハッシュ値（例：short_id）をクライアントに返します。サーバー側の DynamoDB 等に short_id と trace_id の対応表を一時保存しておき、評価時は short_id で受け取って内部で変換します。
メリット: 内部 ID が外部に一切漏れません。
デメリット: 変換用のテーブル管理が発生し、システムが複雑になります。

案2：署名付き評価トークンの発行（JWT 方式）
内容: チャット回答時に、trace_id と有効期限を含めた「署名付きトークン」を発行して返します。評価送信時はそのトークンを添えさせ、サーバー側で署名を検証します。
メリット: ID の偽装や、期限切れの実行に対する評価を完全に防ぐことができます。
デメリット: トークンの生成・検証ロジックの実装コストがかかります。

案3：バックエンドでのセッション紐付け検証
内容: 評価リクエストを受けた際、その trace_id が「現在のリクエストを送ってきたユーザー（session_id 等）」に紐付いているかを DB で突き合わせて確認します。
メリット: 他人のチャット結果に対して評価を送る「横取り」を防げます。
デメリット: 実行ログを検索できる権限が必要になり、評価 API のレスポンスがわずかに遅くなります。
