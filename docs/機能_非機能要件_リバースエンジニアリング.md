# 成果物からの要件リバースエンジニアリング結果

## Phase 1: 成果物の構造分析

成果物を以下の観点で分解し、構成要素を一覧化しました。

| 分析観点 | 内容 |
|---|---|
| **ファイル/モジュール構成** | `template.yaml` (SAMテンプレート), `src/app.py` (Lambdaロジック), `src/requirements.txt` (依存関係) |
| **エントリーポイント** | API Gateway `POST /chat` エンドポイント、Lambda関数 `app.lambda_handler` |
| **データモデル** | LangGraph State (messagesリスト)、DynamoDB (チェックポイント、書き込み履歴) |
| **外部連携** | Amazon Bedrock (Nova Microモデル), Amazon DynamoDB |
| **認証・認可** | API Gateway API Key による認証 (`ApiKeyRequired: true`) |
| **エラーハンドリング** | Lambda内でのtry-except捕捉、HTTPステータスコード(400, 500)の返却、Bedrockクライアントのリトライ設定 |
| **設定・環境変数** | `BEDROCK_MODEL_ID`, `CHECKPOINT_TABLE`, `WRITES_TABLE`, `AWS_REGION` |

## Phase 4: 暗黙要件・欠落要件の指摘

| 指摘ID | カテゴリ | 欠落している要件 | なぜ必要か（理由） | リスク（欠落による影響） | 優先度 |
|---|---|---|---|---|---|
| GAP-001 | セキュリティ | 詳細なレート制限 | 使用プランは定義されているが、具体的なクォータやスロットリング値が明示されていない。 | 低コストモデル(Nova Micro)とはいえ、無制限のリクエストによるコスト増の懸念。 | Medium |
| GAP-002 | 運用性 | 統合監視・通知 | CloudWatch Alarms(エラー率、スロットリング)やSNS通知の設定が欠如している。 | 障害発生時に管理者が即座に検知できない。 | High |
| GAP-003 | 運用性 | 本番用ログ保持期間 | CloudWatch Logsの保持期間が設定されていない(デフォルト無期限)。 | ログストレージコストの増大。 | Low |
| GAP-004 | セキュリティ | ユーザー識別/詳細認可 | APIキーは共有のものが想定されており、ユーザー個別の識別や制限ができない。 | 複数ユーザーがいる場合に、個人を特定した履歴保護ができない。 | Medium |

## Phase 5: 要件カバレッジ 自己検証

| 検証観点 | 結果 | コメント |
|---|---|---|
| 成果物のすべてのファイル/モジュールに対応する要件があるか | ✅ | `template.yaml`, `src/app.py` を完全にカバー。 |
| 機能要件にCRUD操作の漏れはないか | ✅ | DynamoDBへの保存(Create/Update)と取得(Read)は履歴機能としてカバー。削除(Delete)の実装は現時点で見当たらず必要に応じてGAP。 |
| 非機能要件の全カテゴリを検討したか | ✅ | 性能、可用性、セキュリティ、運用性、コストについて抽出。 |
| セキュリティ要件は十分に抽出できたか | ✅ | IAM、API Key、CORS、暗号化を網羅。 |
| 暗黙要件の指摘は行ったか | ✅ | 監視、制限、管理面でのGAPを指摘。 |

**不足がある場合は、追加すべき要件の提案を末尾に記載してください。**
- **履歴削除機能**: 古いチャット履歴をクリーンアップするバッチやAPI機能の検討を推奨。
- **UI連携**: 現在はAPIのみ。Difyや独自UIとの連携要件の定義が必要。
