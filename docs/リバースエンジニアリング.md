## 分析プロセス（この順序で実行してください）

### Phase 1: 成果物の構造分析

成果物を以下の観点で分解し、構成要素を一覧化してください。

| 分析観点 | 内容 |
|---|---|
| **ファイル/モジュール構成** | どのようなファイル・モジュールで構成されているか |
| **エントリーポイント** | 処理の起点は何か（API endpoint / トリガー / 画面） |
| **データモデル** | 扱っているデータの構造・項目・型 |
| **外部連携** | 呼び出している外部サービス・API・ミドルウェア |
| **認証・認可** | 認証方式・権限制御の有無と方式 |
| **エラーハンドリング** | 例外処理・リトライ・フォールバックの実装状況 |
| **設定・環境変数** | 環境依存の設定値・シークレット管理方法 |

### Phase 2: 機能要件の復元

Phase 1の分析結果をもとに、以下の形式で機能要件を復元してください。

| 要件ID | 機能カテゴリ | 機能要件名 | 要件内容（詳細） | 根拠（成果物のどこから読み取ったか） | 確信度 | 備考 |
|---|---|---|---|---|---|---|
| FR-001 | （例: ユーザー管理） | （例: ユーザー登録機能） | （具体的な振る舞いの記述） | （該当コード・設定の箇所を引用） | 確実/推定/不明 | 補足事項 |

#### 確信度の基準

| 確信度 | 定義 |
|---|---|
| **確実** | コード・設定から明確に読み取れる（実装が存在する） |
| **推定** | 実装の断片やパターンから高い確度で推測できる |
| **不明** | 実装が不完全、または意図が複数解釈可能 |

#### 復元時の観点チェックリスト

以下を必ず確認し、該当する機能要件を抽出してください。

- [ ] CRUD操作（作成・読取・更新・削除）の各機能
- [ ] 入力バリデーション（型・桁数・必須/任意・フォーマット）
- [ ] ビジネスロジック（条件分岐・計算・状態遷移）
- [ ] 画面遷移/APIフロー
- [ ] バッチ処理/定期実行処理
- [ ] 通知・メール送信
- [ ] ファイルアップロード/ダウンロード
- [ ] 検索・フィルタリング・ソート
- [ ] 権限別の機能制限

成果物から読み取れる非機能要件を、以下のカテゴリ別に復元してください。

| 要件ID | NFRカテゴリ | 要件名 | 要件内容（詳細） | 根拠（成果物のどこから読み取ったか） | 確信度 | 備考 |
|---|---|---|---|---|---|---|
| NFR-001 | （下記カテゴリ） | （例: APIレスポンスタイム） | （具体的な数値・基準の記述） | （該当コード・設定の箇所を引用） | 確実/推定/不明 | 補足事項 |

#### NFRカテゴリ一覧

| カテゴリ | 復元時の着眼点 |
|---|---|
| **性能** | タイムアウト設定、メモリ割当、同時実行数制限、キャッシュ設定、DBインデックス |
| **可用性** | リトライ設定、ヘルスチェック、Dead Letter Queue、冗長構成 |
| **セキュリティ** | 認証方式、暗号化設定、CORS設定、IAMポリシー、シークレット管理、入力サニタイズ |
| **拡張性** | マイクロサービス分割、キューイング、イベント駆動設計 |
| **運用性** | ログ出力、監視設定、CI/CDパイプライン、IaC管理 |
| **保守性** | コード構造、テストコードの有無、設定の外部化 |
| **互換性** | 対応ブラウザ/OS、APIバージョニング |
| **コスト** | 課金モデル（従量/固定）、リソース上限設定、Reserved/Spot設定 |

※ 成果物に該当しないカテゴリは「該当なし（成果物から読み取れず）」と明記してください。

|---|---|---|

## 出力時の制約

- 「確信度」を必ず付与し、**推測と事実を明確に区別する**こと
- 根拠は「○○ファイルの○○行目」「○○の設定値が○○であること」のように**具体的に引用**すること
- 成果物から読み取れない情報を、根拠なく断定しないこと
- 複数の解釈が可能な場合は、**想定した解釈を明記した上で**要件を記述すること
- 出力量が多い場合は、Phase単位でセクションを分割して出力すること
- 「続きを出力して」で継続可能な構造にすること
- 右記のファイル名で出力してください。/docs/要件.md
