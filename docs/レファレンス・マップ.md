# レファレンス・マップ 履歴

## フェーズ 6: Langfuse による品質監視 (LLMOps)
**更新日**: 2026-04-10

### 1. 機能の入出力マップ (I/O Map)

| 項目 | 入口 (Inlet) | 出口 (Outlet) | 備考 |
| :--- | :--- | :--- | :--- |
| **トレーシング** | Lambda 内の実行データ | Langfuse (Trace / Generation) | 実行の可視化 |
| **コスト管理** | 消費トークン数 | Langfuse コストダッシュボード | 課金の可視化 |
| **文脈管理** | `session_id` | Langfuse (Sessions) | 会話スレッド化 |
| **品質評価** | `trace_id` + `score_value` (0/1) | Langfuse (Scores) | ユーザー満足度の定量化 |
| **機密管理** | **AWS SSM Parameter Store** | Lambda 実行時取得 | 認証情報の秘匿化 (Rule 13) |

### 2. 外部サービスへの依存リスト (Dependency List)

1.  **Langfuse Cloud (SaaS)**:
    - **役割**: プロンプトの実行ログ・コスト・評価を一元管理するプラットフォーム。
2.  **langfuse (Python SDK)**:
    - **役割**: Lambda から Langfuse へデータを安全かつ非同期に送信する。

## フェーズ 5: GitHub Actions (CI/CD)
**更新日**: 2026-04-10

### 1. 機能の入出力マップ (I/O Map)

| 項目 | 入口 (GitHub) | 出口 (AWS/GitHub) | 備考 |
| :--- | :--- | :--- | :--- |
| **トリガーイベント** | `git push`（特定ブランチ） | AWS Resource の更新 | 基本フロー |
| **認証方式** | **OIDC (IDトークン)** | IAM ロールの引き受け | 認証基盤 |
| **脆弱性管理** | GitHub 脆弱性 DB | **Pull Request (自動登録)** | No.5 |
| **手動操作** | GitHub UI ボタン | ワークフロー実行 | No.6 |
| **品質チェック** | Python コード | テスト結果報告 (PR) | No.7 |
| **実行制御** | 後続の実行要求 | 先行ジョブのキャンセル | No.8 |

### 2. 外部サービスへの依存リスト (Dependency List)

1.  **GitHub Actions Runner**:
    *   **役割**: CI/CDワークフローを実行する計算資源。
2.  **SAM CLI**:
    *   **役割**: サーバーレスアプリケーションのビルドおよびデプロイを自動化するツール。
3.  **IAM Role (OIDC用)**:
    *   **役割**: GitHub ActionsがAWSに安全にアクセスするための窓口。

## フェーズ 4-2: Dify 外部ツール連携フェーズ
**更新日**: 2026-04-09

### 1. 機能の入出力マップ (I/O Map)

| 項目 | 入口 (DifyからAPIへ) | 出口 (APIからDifyへ) |
| :--- | :--- | :--- |
| **接続方法** | HTTP POST | JSON（レスポンス） |
| **必要な鍵** | APIキー (x-api-key) | - |
| **渡すデータ** | `message`, `session_id` | `response`, `session_id` |

### 2. 外部サービスへの依存リスト (Dependency List)

1.  **Dify (External Tool)**:
    *   **役割**: APIの「カタログ（スキーマ）」を読み込み、ユーザーの入力を適切な形式でAPIに転送する。
2.  **API Gateway (自作)**:
    *   **役割**: Difyからのリクエストを受け付け、Lambdaを動かす。
3.  **OpenAPI (Swagger) スキーマ**:
    *   **役割**: Difyに対してAPIの使い方を伝えるための「設計図」。

## フェーズ 4-1: API 500エラー解消フェーズ
**更新日**: 2026-04-09

### 1. 機能の入出力マップ (I/O Map)

| 項目 | 入口 (Inlet) | 出口 (Outlet) |
| :--- | :--- | :--- |
| **データ形式** | JSON | JSON |
| **必須項目** | `message`: ユーザーの入力テキスト | `response`: AIの回答テキスト |
| **任意項目** | `session_id`: 会話継続用のID | `session_id`: 入力と同じID |
| **HTTPステータス** | - | **成功**: 200 / **形式エラー**: 400 / **内部エラー**: 500 |

### 2. 外部サービスへの依存リスト (Dependency List)

1.  **Amazon Bedrock (Nova Micro)**:
    *   **役割**: ユーザーのメッセージに対して回答を生成する。
2.  **Amazon DynamoDB**:
    *   **役割**: 会話の履歴を保存・読み出しする。
3.  **IAM (権限管理)**:
    *   **役割**: Lambdaが上記のサービスを操作する許可を与える。
