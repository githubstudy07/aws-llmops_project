# 機能カード QA 履歴

## Phase 4: SAM + API Gateway + LangGraph

### Dify 外部ツール連携 (2026-04-09)
- **Q**: 500エラー（JSON形式不備）の特定方法は？
- **A**: 境界線モニタリングにより、API Gateway と Lambda 間のデータ受け渡し形式の不一致を特定し、JSON 構造を修正することで解決。
- **Q**: API Gateway の URL 末尾のスラッシュの扱いは？
- **A**: 末尾に `/` があると 400 エラーになるため、削除して呼び出すのが正解。

### セキュリティ・認証 (2026-04-09)
- **Q**: API キーをハードコードしてはいけない理由は？
- **A**: GitHub 等への流出による「課金爆弾（不正利用）」のリスクがあるため、環境変数や Secrets で管理すべき。

## Phase 5: GitHub Actions (CI/CD)

### OIDC 連携 & 基本デプロイ (2026-04-10)
- **Q**: AWS の認証キーを GitHub に置かない方法は？
- **A**: OIDC (OpenID Connect) を使い、短命のトークンで AWS ロールを引き受けることで、永続的な鍵を排除。

### セキュリティ高度化 & テスト (2026-04-10)
- **Q**: 機能カードの目的（何を守りたいか）
- **A**: 依存ライブラリの脆弱性の脅威からシステムを守りたい。
- **Q**: レファレンスマップの I/O は追加項目（No.5〜8）に合わせてどう変わるか？
- **A**: 各機能（Dependabot, 手動トリガー等）に応じた入口（UIボタン、脆弱性DB等）と出口（PR、キャンセル等）のデータフローが発生し、これらをレファレンスマップに追記した。

## Phase 6: Langfuse による品質監視 (LLMOps)

### ① トレーシング & オブザーバビリティ (2026-04-10)
- **Q**: このフェーズの目的と懸念点は？
- **A**: LLM の実行過程を可視化し、改善の土台を作ることが目的。懸念点として、Langfuse の具体的な活用方法と、モデル呼び出しにおける正確なコスト把握が挙げられた。
- **Q**: Langfuse はモデル管理用か？ GitHub Secrets ではなく SSM を使う理由は？
- **A**: Langfuse は「実行ログ（ドライブレコーダー）」であり、単なる管理ではなく運用改善が目的。SSM を使うのは、機密情報を AWS 外部（GitHub 等）に出さない「最高レベルのセキュリティ（Rule 13）」を担保するため。

### コスト管理 & SDK 高度化 (2026-04-11)
- **Q**: `CallbackHandler` の初期化で `session_id` がエラーになった原因は？
- **A**: Langfuse SDK v2 以降では、コンストラクタで直接 `session_id` を渡す仕様が廃止（または変更）されたため。最新の推奨手順である `trace` オブジェクトからハンドラーを取得する方法に切り替えることで解決。

### スコアリング（評価） (2026-04-11)
- **Q**: A案（シンプル・ユーザーフィードバック型）を選択した理由は？
- **A**: Dify 等の外部ツールから最小限の負荷で「ユーザーの生の声」をシステムにフィードバックし、改善のサイクル（PDCA）を高速に回すため。
- **Q**: 実装上の技術的ポイントは？
- **A**: `trace_id` を適切に取得・保持し、それに対してスコアを紐付ける仕組みが必要となる。
