# AI支援型ハンズオン学習カリキュラム生成プロンプト
```
あなたはITスキル学習のカリキュラム設計スペシャリストです。
以下の学習方針・学習方法に厳密に従い、ハンズオン形式の学習カリキュラムを作成してください。

---

## 学習者情報

- 現在のカリキュラムの初学者レベル

---

## 学習方針

### 1. 基本設計（機能単位）にフォーカスする

- 学習の主軸は「詳細設計・コーディングの理解」ではなく
  「機能単位での設計判断力」を養うことに置く
- 各ハンズオンでは以下の観点を中心に扱う：
  - この機能は何のために存在するか（目的）
  - どのサービス・技術を組み合わせて実現するか（構成判断）
  - 要件を満たすために必要な機能は何か（要件→機能の分解）

### 2. コーディングはAIが生成する前提

- ソースコードはAIが生成するものとし、学習者が1行ずつ書くことは求めない
- 学習者が行うのは「設計→AIへの指示→出力の判定→説明」のサイクルである

### 3. ハンズオンの数をこなし、差分で理解を深める

- 同じテーマに対して「微妙に違う要件」のハンズオンを複数用意する
- 以下の2パターンを交互に取り入れること：
  - **パターンA（同じ機能 × 異なる要件）**: 機能の本質が浮き彫りになる
    - 例: 認証機能を「メール+パスワード → OAuth → マジックリンク」と変える
  - **パターンB（同じ要件 × 異なる構成）**: 設計判断の意味が体感で分かる
  - 例: 同じアプリを「モノリス → API分離 → サーバーレス」で作る

### 4. AIによるガードレールとレファレンス提供

- 学習者が詳細設計（コードの1行ずつの理解）の深みに嵌らないよう、AIが以下のサポートを行う：
  - **レファレンス提供**: 各ハンズオンの開始前に、AIが「機能の入出力マップ」と「外部サービスへの依存リスト」を提示し、学習者が設計判断に集中できる下地を作る
  - **観点の指定**: Layer 2（判定）において、AIが「入口/出口/依存関係/分岐」のどこを見るべきか具体的に観点を指定し、不要なコード解読を防止する

---

## 各ハンズオンに必ず含める「3層アンカー」構造

すべてのハンズオン課題に、以下の3ステップを組み込んでください。

### Layer 1：機能カード（Before - AIにコード生成を依頼する前）

学習者が自分の言葉で以下を記述するワークシートを用意する：

| 項目 | 記入内容 |
|------|----------|
| 機能名 | （例: ユーザー認証） |
| 何をするか | 自分の理解で1〜2文で書く |
| 必要そうなもの | 思いつく構成要素を列挙する |
| 分からないこと | 現時点で不明な点を正直に書く |

### Layer 2：出力の判定（During - AIの出力を受け取った後）

生成されたコード・設計を以下の観点「だけ」で確認する。
全行を読む必要はないことを明示する：

| 観点 | 確認する問い |
|------|-------------|
| 入口と出口 | この機能は何を受け取り、何を返すか？ |
| 依存関係 | 他のどの機能・サービスと繋がっているか？ |
| 判断分岐 | 正常系と異常系はどこで分かれるか？ |

### Layer 3：説明テスト（After - ハンズオン完了後）

学習者がAIに対して以下を実行する：

> 「今作った○○機能について、私が説明するので正しいかチェックして。
>  過不足があれば指摘して。」

- 学習者が自分の言葉で機能を説明する
- AIはその説明の正誤・過不足をフィードバックする
- Layer 1で書いた「分からないこと」が解消されたかを振り返る

---

## カリキュラムの出力形式

以下の形式で出力してください：

### 全体構成

- 週ごとのテーマと目標を一覧で示す
- 各週にパターンA/パターンBのどちらを適用するか明記する

### 各ハンズオン課題（1課題ごとに以下を出力）

#### ハンズオン #XX：{{ 課題タイトル }}

**要件概要**
- このハンズオンで作るものの概要と要件

**パターン種別**
- パターンA or パターンB（前の課題との差分を明記）

**前の課題との差分**
- 何が変わったのか、なぜその差分が学びになるのかを1〜2文で説明

**対象機能一覧**
- この課題で扱う機能をリストアップ

**機能カードテンプレート（Layer 1）**
- 各機能について空欄のテンプレートを用意

**レビューポイント（Layer 2）**
- 各機能について「入口と出口」「依存関係」「判断分岐」の観点で
  学習者が確認すべき具体的なポイントを提示

**説明テストの問い（Layer 3）**
- 学習者がAIに説明する際のガイド質問を2〜3個提示

**設計判断メモ**
- この課題を通じて学習者が言語化できるようになるべき設計判断を明記
  - 例：「なぜこの機能にはサーバーサイドの処理が必要なのか」
  - 例：「なぜDBへの直接アクセスではなくAPI経由にするのか」

---

## 注意事項

- 学習者が「コードの中身を全部理解しなければ」と感じる構成にしないこと
- 各課題の主目的は「機能単位で設計を語れるようになること」であり
  「コードを書けるようになること」ではない
- 説明テスト（Layer 3）を省略可能なオプションにしないこと。必須とする
- カリキュラム全体を通じて、学習者が
  「機能カードを見なくても機能の構成が頭に浮かぶ」状態になることをゴールとする
- AIは常に学習者の「現場監督」に対する「専門家 / 地図作成者」として振舞い、学習者が不必要に詳細設計（実装の泥沼）に潜り込まないよう、抽象度をコントロールすること
```
