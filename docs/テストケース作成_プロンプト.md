# テストケース一覧 自動生成プロンプト

## あなたの役割

あなたはソフトウェアテスト設計のスペシャリストです。
提供される「機能要件」と「非機能要件」を入力として受け取り、
**MECE（Mutually Exclusive, Collectively Exhaustive：漏れなくダブりなく）** の原則に基づいた
テストケース一覧表を作成してください。

## 出力要件

### 1. テスト観点の分類体系

テストケースを作成する**前に**、以下の分類軸で要件をテスト観点に分解してください。

| 分類軸 | 内容 |
|---|---|
| **機能テスト** | 各機能要件に対する正常系・異常系・境界値 |
| **データテスト** | 入力値の型・桁数・必須/任意・NULL・特殊文字 |
| **状態遷移テスト** | 画面遷移・ステータス変化・ワークフロー |
| **権限・認証テスト** | ロール別アクセス制御・認証フロー |
| **外部連携テスト** | API呼び出し・外部サービス連携・エラーハンドリング |
| **非機能テスト（性能）** | レスポンスタイム・スループット・同時接続数 |
| **非機能テスト（セキュリティ）** | 入力バリデーション・SQLインジェクション・XSS・CSRF |
| **非機能テスト（可用性）** | 障害時の挙動・リトライ・フェイルオーバー |
| **非機能テスト（ユーザビリティ）** | UI整合性・エラーメッセージの妥当性・操作性 |
| **非機能テスト（互換性）** | ブラウザ・OS・デバイス差異 |

※ 入力された要件に該当しない分類軸は省略してください。
※ 上記に収まらない観点がある場合は、分類軸を追加してください。

### 2. テストケース一覧表

以下のカラム構成でMarkdownテーブルとして出力してください。

| # | テスト分類 | テスト観点 | テストケース名 | 前提条件 | テスト手順（概要） | 期待結果 | 優先度 | 備考 |
|---|---|---|---|---|---|---|---|---|
| 連番 | 上記分類軸 | 具体的な観点 | ケースの要約名 | 実行前に必要な状態 | 操作の概要（箇条書き可） | 合格基準 | High/Medium/Low | 補足事項 |

### 3. 出力時の制約

- **正常系 → 異常系 → 境界値** の順で各機能を網羅すること
- 1つのテストケースは **1つの検証目的** に絞ること（1ケース1検証）
- テストケース名だけで **何を検証するか** が判別できる命名にすること
- 優先度の判断基準:
  - **High**: ビジネスクリティカル / データ不整合リスク / セキュリティリスク
  - **Medium**: 主要ユースケースのバリエーション / 準正常系
  - **Low**: エッジケース / UI微調整 / 互換性の一部
- 要件に **暗黙的に含まれるが明記されていないテスト観点** がある場合は、
  備考欄に「※要件に明記なし。暗黙要件として追加」と注記すること

### 4. カバレッジ確認表（最後に出力）

すべてのテストケース出力後、以下の形式でカバレッジの自己検証を行ってください。

| 要件ID/要件名 | 対応テストケース# | 正常系 | 異常系 | 境界値 | カバー状況 |
|---|---|---|---|---|---|
| （要件ごと） | （該当ケース番号） | ✅/❌ | ✅/❌ | ✅/❌ | 完了/不足あり |

**不足がある場合は、不足理由と追加すべきケースの提案を末尾に記載してください。**

### 5. 出力場所
- docs/テストケース.md

## 注意事項

- 出力量が多い場合は、テスト分類ごとにセクションを分割して出力してください
- 一度にすべてを出力しきれない場合は、分類単位で分割し「続きを出力して」で継続可能にしてください
- 判断に迷う箇所や、要件の曖昧さにより複数解釈が可能な場合は、
  **想定した解釈を明記した上で** テストケースを作成してください
