# AI実装の鉄則：デバッグ泥沼回避ガイド (現場監督用)

このドキュメントは、AI職人が「解決できないエラー」にハマり、時間を浪費するのを防ぐための「現場監督」向け管理マニュアルです。実装フェーズに入る前に、必ずAIにこの内容を再読させてください。

## 🚨 AIの「沼」への予兆 (レッドフラグ)
AIの回答やコードに以下の傾向が見えたら、即座に作業を中断させ、「仕切り直し」を命じてください。

1. **力技（Brute Force）の出現**:
   - 本来のライブラリの仕様を無視し、ディレクトリを物理走査（`os.walk`等）してファイルを探し始める。
   - インポートエラーを回避するために、実態のない「Mockクラス」を大量に作り始める。
2. **コードの複雑化**:
   - 直すたびにコードが肥大化し、`try-except` が何重にも重なる。
   - 正解は常にシンプルであるはずだという視点を失っている。
3. **同じエラーの繰り返し**:
   - 「次こそは大丈夫です」と言いながら、同じエラーメッセージ（例：ImportError）で3回以上デプロイ・起動に失敗する。

## 🛠️ 泥沼を回避する「現場監督」の命令集

AIがハマりかけた時、あるいは実装を開始する前に、以下のコマンドを投げてください。

### 1. 「最新仕様を検索せよ (Web Search Enforcement)」
> 「独力で解決しようとせず、必ず最新の [ライブラリ名/サービス名] の公式ドキュメント（vX.X等、最新バージョン）をWeb検索して、[機能名] の正しい実装パスと初期化方法を確認してから提案して。」

### 2. 「依存関係を疑え (Check requirements.txt)」
> 「コード自体ではなく、`requirements.txt` やデプロイ環境に必要なパッケージが不足していないか？ バージョン競合が起きていないか、まずそこから確認して。」

### 3. 「最小構成での検証 (Minimal Reproduction)」
> 「現在の肥大化した `src/app.py` ではなく、[問題の機能] だけを動かすための最小限のテストコード（Scaffold）を作成して、そこで動作を切り分けて。」

### 4. 「クリーンアップとリビルド (Clean Start)」
> 「これまでのデバッグ用のゴミコードを一度すべて削除して、公式の公式サンプルに基づいた『最もシンプルで標準的な構成』に戻してから、ゼロベースで組み直して。」

## 💡 現場監督の心得
AIは「手持ちの古い知識」で無理やり問題を解こうとする習性があります。**「15分粘って解決しないなら、知識（ドキュメント）をアップデートさせる」**ことが、プロジェクトを成功に導く最短ルートです。
