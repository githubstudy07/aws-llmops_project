## AWS IAM (Identity and Access Management)

### **Q: インラインポリシーが複数セットされた場合、優先度はどう計算される？**
- **対象機能**: IAM ポリシー評価ロジック (Evaluation Logic)
- **AIの解説(回答)**: 
    - **理由**: IAM では同じ種類のポリシー（インラインポリシー同士など）に「順番」や「優先順位」はありません。
    - **ルール**: すべてのポリシーは「合算」され、以下の順序で判定されます。
        1. **Explicit Deny (明示的な拒否)**: 1つでもあれば即拒否。最強のルール。
        2. **Explicit Allow (明示的な許可)**: Deny がなく、1つでも Allow があれば許可。
        3. **Default Deny (デフォルト拒否)**: 上記以外はすべて拒否。
    - **運用のアドバイス**: インラインポリシーを細かく分けても、全体として「許可の足し算」になるだけなので、管理しやすい単位で分割して問題ありません。

## Amazon Bedrock / Nova

### **Q: Nova Micro は東京リージョンでそのまま呼び出せる？クロスリージョン推論とは？**
- **対象機能**: Nova Micro モデルの呼び出し (Cross-Region Inference)
- **AIの解説(回答)**: 
    - **理由**: 2026年時点で Nova Micro は特定のリージョンでホストされていますが、東京リージョン (`ap-northeast-1`) からも「推論プロファイル」を使用することで、低レイテンシかつリージョンを意識せずに呼び出し可能です。
    - **設定項目**: モデル ID として `apac.amazon.nova-micro-v1:0` を使用します。
    - **注意点**: IAM 権限に `bedrock:InvokeModel` だけでなく、プロファイルのリソースに対する権限も必要になる場合があります。

## LangGraph

### **Q: LangGraph の主要な概念（State, Nodes, Edges）とは何？**
- **対象機能**: グラフベースのエージェント構築
- **AIの解説(回答)**: 
    - **State (状態)**: グラフ全体で共有されるデータ構造（辞書形式）。
    - **Nodes (ノード)**: 実際の処理（AIの呼び出しやツールの実行）を行う関数。
    - **Edges (エッジ)**: ノード間の遷移ルール（次にどのノードに進むか）。

## DynamoDB / 永続化 (Persistence)

### **Q: 会話履歴の保存で既存テーブルと衝突したり、権限エラーが出た場合の対処は？**
- **対象機能**: DynamoDB を使用したチェックポインター (永続化)
- **AIの解説(回答)**: 
    - **理由**: `LangGraph` の標準的なテーブル名（`langgraph-checkpoints`）は既に他プロジェクトで使用されている可能性があり、スキーマが異なるとエラーになります。
    - **設定項目**: ハンズオン専用の接頭辞（例: `handson-`）を付けたテーブル名を使用します。
    - **権限**: `DynamoDBSaver` の内部でデータの一括書き込み・読み込みを行うため、`dynamodb:BatchWriteItem` および `dynamodb:BatchGetItem` の権限が必須です。

## 課金・コスト管理

### **Q: テスト実行や中断中の放置でどのくらいの費用がかかる？**
- **対象機能**: コスト管理 (AWS Billing)
- **AIの解説(回答)**: 
    - **理由**: 今回使用する Nova Micro や Haiku は極めて安価なモデルであり、DynamoDB も「オンデマンド」モードに設定しているため、未使用時の基本料金は発生しません。
    - **数値**: Haiku 1回の呼び出しは約 0.002円 未満。DynamoDB のデータ保持（数KB程度）は無料枠内に収まります。

## AWS SAM / API Gateway

### **Q: Lambda 関数のソースコードを src/ ディレクトリに分けるのはなぜ？**
- **対象機能**: SAM のプロジェクト構成 (`CodeUri`)
- **AIの解説(回答)**: 
    - **理由**: SAM CLI でビルド（`sam build`）する際、`CodeUri` で指定したディレクトリ以下のファイルがパッケージ化されます。
    - **メリット**: 
        1. **不要なファイルの排除**: プロジェクトルートにあるテストスクリプトやドキュメントを Lambda パッケージに含めないことで、デプロイ容量を削減できます。
        2. **依存管理**: `src/` 配下に `requirements.txt` を置くことで、関数に必要なライブラリだけをビルド時にインストールさせることができます。
    - **注意点**: ハンドラーの指定は、`src/` をルートとした相対パス（例: `app.lambda_handler`）になります。

### **Q: AWS SAM とは何？このハンズオンでの使用の「肝」は？**
- **対象機能**: AWS Serverless Application Model (SAM)
- **AIの解説(回答)**: 
    - **SAM とは**: AWS 上にサーバーレスアプリケーション（Lambda, API Gateway, DynamoDB 等）を構築するためのフレームワークです。
    - **用途**: 
        1. **「コードでの定義 (IaC)」**: インフラ構成を YAML ファイルで管理し、手動操作を排除します。
        2. **「一括デプロイ」**: プログラムコードのアップロード、API 構築、権限設定を 1 コマンドで実行します。
    - **ハンズオンにおける「肝」**:
        1. **「個人開発からプロダクトへ」**: これまでの「ローカルでスクリプトを動かす」段階から、世界中から呼び出し可能な「Web API システム」へと昇華させる点にあります。

## トラブルシューティング記録（Phase 4 デプロイ）

### **Q: SAM デプロイ時に 502 Bad Gateway が発生し、ログに "No module named 'langgraph'" と出る理由は？**
- **原因**: `sam package` 実行時に、ビルド済みの成果物（`.aws-sam/build/` 内）ではなく、ソース元の `src/` フォルダを指定してパッケージ化してしまい、外部ライブラリが欠落したため。
- **解決策**: 手動で `.aws-sam/build/LangGraphAgentFunction/` の中身を zip 化し、S3 にアップロードしてデプロイすることで確実にライブラリを同梱させました。

### **Q: デプロイ時に [AWS::EarlyValidation::ResourceExistenceCheck] で失敗するのはなぜ？**
- **原因**: 以前構築した際のスタック（AWS SAM CLI の管理用スタック等）が `DELETE_FAILED` などの不安定な状態で残っており、新しいデプロイをブロックしていたため。
- **解決策**: `--resolve-s3` に頼らず、独自の S3 バケットを新規作成し、スタック名も刷新して干渉を回避しました。

### **Q: 400 Bad Request (ログにエラーなし) が返ってきた際のチェックポイントは？**
- **原因**: API Gateway の URL 末尾に不要な `/`（スラッシュ）が付いていると、リクエストが正しく Lambda へルーティングされず、秒速で 400 エラーを返すことがあります。
- **解決策**: URL 末尾のスラッシュを削除して呼び出す（例: `/Prod/chat/` → `/Prod/chat`）。

## AI駆動開発の進め方・管理手法

### **Q: AIが自動で行った作業の中身が「ブラックボックス」になるのを防ぐには？**
- **対象機能**: 開発プロセス管理 (Accountability)
- **AIの解説(回答)**: 
    - **悩み**: ソースコードを全て読もうとすると AI 駆動のメリットが消え、読まないと中身が分からず「AIの言われるがまま」になるジレンマ。
    - **解決策**: **「現場監督（ユーザー）vs 熟練職人（AI）」** という役割分担を定義します。ソースコード（Syntax）ではなく、仕様（Spec）と証拠（Proof）で管理します。
    - **具体的な手法**:
        1. **[機能要件]と[非機能要件]を分離**: 「何ができるか」だけでなく「安全性・コスト・運用性」を管理項目にする。
        2. **チェックリストの導入**: [Manager_Checklist.md](file:///C:/Users/naoji/Documents/projects/aws-llmops_project/docs/Manager_Checklist.md) を作成し、各フェーズの完了条件を明文化する。
        3. **証跡（エビデンス）の義務化**: AIに「完了」を報告させる際、必ずテストログや設定の根拠を提示させる。
    - **メリット**: 現場監督は「ステークホルダー（上長や顧客）への説明責任」を果たすことに集中し、AIは「実装の細部」に集中できます。

## セキュリティと認証 (Phase 4 強化)

### **Q: API 認証 (API Key 等) の実装が後回しになりがちなのはなぜ？対策は？**
- **対象機能**: 認証セキュリティ (Non-functional requirement)
- **AIの解説(回答)**: 
    - **理由**: 開発初期は「まず動くこと（機能要件）」に集中するため、認証などの「守り（非機能要件）」が意識から漏れやすいためです。
    - **対策**: `Manager_Checklist.md` に「安全性 (Security)」のカテゴリを設け、実装開始前に必須項目として定義しておくことで、職人(AI)の意識漏れを防ぐことができます。

### **Q: API キーをドキュメントやコードに「生」で書いてはいけない理由は？**
- **理由**: 
    1. **流出リスク**: GitHub 等にコミットされると、ボットによって数秒で検知・悪用されます。
    2. **課金リスク**: 第三者が API を悪用し、膨大な LLM 使用料（Nova Micro 等）がアカウント所有者に請求される「課金爆弾」に繋がります。
- **対策**: キーは AWS Secrets Manager や GitHub Secrets で管理し、ドキュメント上ではマスク（伏せ字）または「管理場所への参照」に留めます。

### **Q: SAM で API Key を使う際、「明示的リソース定義」が推奨される理由は？**
- **理由**: SAM の「暗黙的な API 生成」は手軽ですが、API Key や使用計画（Usage Plan）などの複雑な依存関係が必要な場合、生成オプションが限定され、デプロイエラー（Stage not found 等）が起きやすいためです。
- **解決策**: `AWS::ApiGateway::RestApi`, `Deployment`, `Stage` を明示的に定義することで、リソース作成の順番（依存関係）を確実にコントロールできます。
