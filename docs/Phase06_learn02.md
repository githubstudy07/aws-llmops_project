# 実務レベルのデプロイ・セオリー (Inner Loop vs Outer Loop)

実務においては、**「CI/CD パイプラインを汚さない・無駄打ちしない」** ことが鉄則です。そのため、以下の「手元の検証（Inner Loop）」と「自動化された検証（Outer Loop）」を使い分けるのが正解となります。

## 1. 何よりも先に「ローカルでのユニットテスト」 (MUST)
* **理由**: 基本的なシンタックスエラーやロジックのミス（例：変数の未定義など）を GitHub にプッシュした後にCI上で気づくのは、CIの実行時間（課金・リソース）の無駄であり、職人として恥ずかしい行為とされます。
* **アクション**: `pytest` を実行し、コードの内部ロジックが壊れていないことを数秒で確認します。

## 2. 「SAM Build / Validate」による構文チェック (SHOULD)
* **理由**: レポーティングツールや CloudFormation (`template.yaml`) のインデントミス、リソース間の参照不備は、AWS へのデプロイの最終段階でロールバックを引き起こします。GitHub Actions を走らせる前に、手元で `sam build` が通ることを確認するのが「手戻り」を防ぐプロの作法です。

## 3. GitHub Actions 経由のデプロイ (BEST)
* **理由**: 手元の環境（OSやライブラリの実パス等）に依存せず、**「誰がやっても同じ結果になる（再現性）」** ことを保証するためです。
* **アクション**: ローカルテストと構文チェックが通ったら `git push` し、CI/CD パイプライン上のセキュリティスキャンや自動化テストを一丸として経て、クラウド本番環境へ安全に反映させます。
