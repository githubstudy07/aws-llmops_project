# AWS LLMOps ユースケース・シナリオ・理由 記録

## Phase 8: コンテナデプロイと環境最適化

### Phase 8-2 | GCC バージョン不足の解消 (Lambda Python 3.12 移行)
#### 1. どのような要件か（ユースケース）
「CrewAI や NumPy などの高度な数値計算・マルチエージェントライブラリを Lambda (Container) 上で動かす際に、**OSレベルのコンパイラ (GCC) のバージョン不足によるビルドエラーを解消し、安定した実行環境を構築したい**」という要件。

#### 2. 想定シナリオ
Phase 8-1 のデプロイにおいて、Amazon Linux 2 ベースの Python 3.11 イメージを使用していたが：
1.  **問題発生**: `numpy` の最新版がビルド時に GCC 9.3 以上を要求。
2.  **失敗**: Amazon Linux 2 の標準 GCC は 7.3.1 であり、ビルドが停止。
3.  **解決策**: Amazon Linux 2023 ベースの Python 3.12 公式イメージに切り替えることで、標準で GCC 11 以上を利用可能にし、ビルドを成功させる。

#### 3. なぜこの機能が必要か（理由）
*   **依存関係の解決**: 最新の AI 関連ライブラリはより新しい OS/コンパイラを前提としていることが多いため、実行環境を最新化（AL2023）する必要がある。
*   **メンテナンス性**: Lambda のカスタムコンテナ環境において、手動で GCC をアップグレードするよりも、公式の最新ベースイメージを利用する方が安全で標準的であるため。

### Phase 8-2.1 | 500エラーの隔離と最小構成検証
#### 1. どのような要件か（ユースケース）
「デプロイ後に HTTP 500 エラーが発生し、CloudWatch ログすら出力されないような重度の障害が発生した際、**正常に動作する最小単位まで機能を削ぎ落とすことで、原因がプラットフォーム設定にあるのか、特定ライブラリのインポートにあるのかを突き止めたい**」という要件。

#### 2. 想定シナリオ
1.  **問題発生**: Phase 8-2 のデプロイ後、API 呼び出しが 500 エラーで失敗。原因が NumPy のインポートか、Lambda の起動設定（CMD/ENTRYPOINT）か不明。
2.  **実行**: `src/app_crew.py` を「ライブラリのインポートを handler 内で try-except 実行する診断モード」に書き換え、外部依存による起動失敗を回避する。
3.  **判定**: もしこれで 500 が直れば「ライブラリのインポート」が原因。直らなければ「Lambda/Docker の設定」が原因。

#### 3. なぜこの機能が必要か（理由）
*   **デバッグの迅速化**: 複雑な依存関係を持つ CrewAI 環境では、どこでエラーが起きているかを推測するよりも、正常に動く「最小構成」から一歩ずつライブラリを戻していく方が、結果として最短で解決できるため（デバッグ泥沼回避ガイドの鉄則）。
*   **インフラ健全性確認**: そもそも API Gateway と Lambda の紐付けや、コンテナの起動自体が成功しているかを確実にするため。

## Phase 7: CrewAI による自律型エージェント (広告分析・コピー作成)

### Phase 7-1 | 広告キャッチコピー作成の自動化（初学者向け：リサーチ ＆ ライティング）
#### 1. どのような要件か（ユースケース）
「特定の商品について、**市場のトレンドやユーザーの声を調査し、それを踏まえた効果的な広告キャッチコピーを複数パターン自動生成したい**」という要件。複数の専門的な役割（エージェント）を連携させることで、単一のプロンプトでは難しい多角的な思考が必要なタスクを自動化します。

#### 2. 想定シナリオ
新しい「AI搭載の健康飲料」を売り出したいマーケティング担当者が：
1.  **リサーチャー（エージェント1）**: SNSやWebサイト上の「健康飲料に対する悩みやトレンド」を調査し、ターゲットに刺さる「キーワード」を抽出する。
2.  **コピーライター（エージェント2）**: リサーチャーの報告を元に、商品の特徴を活かしたキャッチコピーを3案作成する。
3.  **結果**: 網羅的な調査結果と、根拠のある広告コピーがセットで得られる。

#### 3. なぜこの機能が必要か（理由）
*   **役割の分離（専門性）**: 「調べる力」と「書く力」を分けることで、プロンプトが巨大化・複雑化するのを防ぎ、出力品質を安定させるため。
*   **自律性の導入**: 手順を細かく指示するのではなく、「目標（Goal）」を与えるだけでAIが自ら最適な成果物を生成する体験を得るため。
*   **LLMOps の実戦**: 複数のエージェントが動く際の「コストの不透明さ」を Langfuse で可視化・管理し、ビジネスで使えるエージェント運用の勘所を掴むため。


## Phase 6: Langfuse による品質監視 (LLMOps)

### Langfuse-④ | データセット & 実験
#### 1. どのような要件か（ユースケース）
LLM アプリケーションの運用において、**「プロンプトを改善したつもりが、別のパターンで回答が悪化した（デグレード）」という事態を防ぎ、改善の効果を数値で客観的に証明したい**という要件。

#### 2. 想定シナリオ
ユーザーからの「もっと親しみやすい口調にしてほしい」というフィードバックに対し、開発者がシステムプロンプトに「ユーモアを交えて回答して」と追記した場合：
*   **懸念点**: これにより、エラー報告などの「真面目さが求められる回答」まで冗長になったり、誤った情報を冗談で返したりする副作用が発生する可能性がある。

#### 3. なぜこの機能が必要か（理由）
*   **主観の排除**: 「良くなった気がする」という感覚ではなく、「過去の 10 件の成功パターンにおいて、新プロンプトでも 100% 同等以上の品質を維持できているか」を定量評価するため。
*   **アジャイルな改善**: 修正による影響範囲を恐れずに、データに基づいて「この改善は安全である」と確信を持って本番反映（CI/CD）へと繋げるため。
*   **LLMOps の PDCA 完遂**: 監視（Trace）で得た「生の声」をデータセット化し、次の改善の「種」として活用する、LLMOps の循環サイクルを完成させるため。

---

### Langfuse-④ | Baseline Run（基準実行）

#### 1. どのような要件か（ユースケース）
プロンプトを修正する前に、**「今のシステムのままだと、このデータセットに対してどのような結果を返すのか」という現状のスコアを測定・固定したい**という要件。

#### 2. 想定シナリオ
データセットに 3 件の問題を登録した場合：
*   **現状**: 今の古いプロンプトで実行した結果、正解（Expected Output）との一致度や内容の正確性が、人間から見て「100点満点中 60点」だったと記録する。これが **Baseline（基準点）** になる。

#### 3. なぜこの機能が必要か（理由）
*   **変化の可視化**: 基準点がないと、後にプロンプトを「改善」した際、スコアが向上したのか悪化したのかを客観的に判断できないため。
*   **公平な比較**: 全く同じ入力データを使い、現行のコード環境で一斉テストを行うことで、比較の公平性を担保するため。

### Langfuse-⑤ | プロンプト観測性の強化（案A）

#### 1. どのような要件か（ユースケース）
「プロンプトを修正した後の記録」において、**「そもそもあの時、どんなシステムプロンプトを使っていたのか？」を Langfuse の画面だけで完結して確認したい**という要件。

#### 2. 想定シナリオ
データセットを用いた実験（Run）の結果を比較した際、「回答が変わったのはわかったが、指示（System Prompt）がどう変わったから結果が良くなったのか」を、外部のコードを遡ることなく一目で左右比較できるようにする。

#### 3. なぜこの機能が必要か（理由）
*   **デバッグ効率の向上**: ソースコードの履歴を遡らなくても、過去のあらゆるトレースの「前提条件（指示内容）」がログとして厳密に残るため。
*   **比較機能のフル活用**: Langfuse の比較モードにおいて、新旧のプロンプト文面を左右に並べて視覚的に差分を確認するため。

---

### Phase 6-6 | データセット & 実験 (再開セッション)

#### 1. どのような要件か（ユースケース）
**「プロンプト改善時の副作用検知と品質の定量評価」**
実運用で蓄積されたユーザーとの会話データ（トレース）から、特に重要なケースや失敗したケースを「テストケース」として抽出します。これに対し、プロンプトを修正した新バージョンが「以前より良くなったか？」「他の重要な回答を改悪させていないか？」を機械的に一括実行して比較したいという要件。

#### 2. 想定シナリオ
1.  **データ抽出**: 運用中に「Good」評価がついた回答や、逆に「Bad」だったが改善したい入力を Langfuse 上でデータセット `handson-dataset-01` として保存する。
2.  **Baseline（基準）の測定**: 現行のプロンプトでデータセットを実行し、出力結果を保存する（`baseline-run`）。
3.  **プロンプト改善**: システムプロンプトに「専門家として回答せよ」等の指示を追加・修正する。
4.  **比較実験**: 修正後のプロンプトで再度データセットを実行し（`improved-run`）、新旧の回答を Side-by-Side で比較・評価する。

#### 3. なぜこの機能が必要か（理由）
LLMのプロンプト変更は「あちらを立てればこちらが立たず」の状況に陥りやすいため、勘や経験ではなく、過去の成功/失敗データに基づいた**「回帰テスト」**を自動化することで、デプロイの心理的ハードルを下げ、継続的な改善サイクル（PDCA）を回すため。

---

### プロンプトデプロイの実務セオリー (LLMOps)

#### 1. どのような要件か（ユースケース）
実験で品質向上が確認されたプロンプトを、**「安全に、かつ迅速に本番環境へ反映・デプロイしたい」**という要件。

#### 2. 想定シナリオ（3つのセオリー）
*   **パターンA：コードベース管理**
    *   `src/app.py` などのソースコード内のプロンプトを直接書き換え、GitHub Actions 等の CI/CD パイプラインでデプロイする。
    *   *理由*: バージョン管理の整合性が完璧。
*   **パターンB：外部設定管理 (Config)**
    *   AWS SSM パラメータストアや AppConfig にプロンプトを保存し、Lambda が起動時に取得する。
    *   *理由*: コードの再ビルドなしで即時反映が可能。
*   **パターンC：プロンプト管理ツール活用（推奨）**
    *   Langfuse 等のツール上でプロンプトを管理し、`production` 等のタグを付け替えることで本番を切り替える。
    *   *理由*: 非エンジニアによる頻繁な調整を可能にし、実験から適用までのサイクルを最短化するため。

#### 3. なぜこの機能が必要か（理由）
プロンプトはアプリケーションの「モデルの振る舞い」そのものを決定する最重要資産であり、その更新プロセスをシステム化（LLMOps）することで、手作業によるミスを防ぎ、継続的な品質改善を可能にするため。

---

### Phase 6-7 | Langfuse プロンプト管理 (Prompt Management)

#### 1. どのような要件か（ユースケース）
**「プロンプト管理のコードからの分離と、ノーコード・デプロイの実現」**
プロンプトをソースコードから完全に切り離し、Langfuse 上で一元管理したい。プロンプトの修正時にコードの変更やデプロイを待つことなく、管理画面の操作だけで即座に本番環境の挙動を切り替えたいという要件。

#### 2. 想定シナリオ
1.  **プロンプト登録**: Langfuse の「Prompts」メニューにて、実験で良好だった専門家プロンプトを名前（例: `main-chatbot-prompt`）を付けて保存する。
2.  **タグ付け**: 保存したプロンプトに `production` というラベルを付与する。
3.  **SDK 連携の実装**: `src/app.py` を修正し、プロンプトを直書きせず Langfuse から動的に取得するコードに変更する。
4.  **動作検証**: 管理画面のタグ付け替えのみで回答が変わることを確認する。

#### 3. なぜこの機能が必要か（理由）
プロンプトの改善速度を上げるため。実務における「朝に実験して、昼には本番に反映」という高速な PDCA サイクルを、バージョニングされた安全な状態で実現するため。

### Phase 8-2.2 | Lambda CLI直接Invoke & CloudWatch診断
#### 1. どのような要件か（ユースケース）
「API Gateway 経由では詳細なエラーが隠蔽され、さらに CloudWatch Logs にログが生成されないという最悪の状況（サイレント・エラー）において、**AWS CLI を用いて直接 Lambda を叩き出す（Invoke）ことで、APIGWの層を迂回してランタイム固有の起動失敗ログを強制的に取得したい**」という要件。

#### 2. 想定シナリオ
1.  **問題発生**: 最小構成コードをデプロイしても 500 エラーが継続。APIGW のログも CloudWatch の Lambda ログが見つからない。
2.  **実行**: ローカル（または作業環境）のターミナルから `aws lambda invoke` コマンドを実行し、Lambda のレスポンスペイロードおよび `LogResult` をキャプチャする。
3.  **分析**: 取得した `LogResult` に含まれる Python ランタイムのエントリーポイント読み込み失敗（ImportError や SyntaxError 等）を確認する。

#### 3. なぜこの機能が必要か（理由）
*   **APIGW 隠蔽の打破**: APIGW は「何か失敗した」ことは教えてくれるが「コンテナの中で何が起きて 500 になったか」は教えてくれない。直接 Invoke により情報を「引きずり出す」必要があるため。
*   **Init層エラーの捕捉**: CloudWatch Logs は Lambda の Init フェーズが成功し、実行フェーズに入らないと記録されない場合がある。CLI の `LogResult` はこの Init フェーズの死に際を確認する唯一の手がかりになるため。

### Phase 8-2.3 | API Gateway の強制再デプロイによる統合不一致の解消
#### 1. どのような要件か（ユースケース）
「AWS Lambda の PackageType を Zip から Image に変更するなど、**統合先の定義に重大な変更を加えたが、API Gateway のデプロイメントが更新されていないために古い設定で呼び出しが発生し 500 エラーになっている状況を解消したい**」という要件。

#### 2. 想定シナリオ
1.  **問題発見**: Lambda 側はコンテナ化されているが、APIGW の最終デプロイ日が Lambda の変更日より前であることを特定。
2.  **実行**: SAM テンプレート内の `AWS::ApiGateway::Deployment` リソースの論理 ID をインクリメント（V2 -> V3）し、強制的に新規デプロイメントを作成・適用させる。
3.  **検証**: デプロイ完了後、APIGW が新しいコンテナ版 Lambda を正しく Invoke できるようになることを確認する。

#### 3. なぜこの機能が必要か（理由）
*   **不変リソースの罠回避**: 標準的な CloudFormation の挙動では、APIGW の Deployment は「プロパティ」が変わらない限り再作成されない（再適用されない）。論理 ID の変更は、SAM において確実に最新の状態をステージに反映させるための正攻法であるため。
